# Avery Castillo — Starter Snapshot

The "before" state used to record the launch walkthrough. Copy `starter/my-documents/` into a fresh test folder and the demo runs end-to-end against it.

The committed end-state lives one level up at [`examples/avery-castillo/my-documents/`](../my-documents/) — that's what the workflow produces once every beat has run.

**What's in here:**

- `my-documents/resume.md` — Avery's source resume (v3). Contains the unsupported "$4.2M in pipeline" claim that `claim-check` will flag in Beat 5.
- `my-documents/story-bank.md` — three STAR+R stories (Docker demo refactor, PoC restructuring, Snowflake integration guide).
- `my-documents/applications.md` — empty tracker table. The Polaris Data row gets created during the demo.
- `my-documents/proof-assets/` — two proof-asset briefs (demo-environment refactor, Snowflake integration guide). Pre-existing Avery work, not generated by the walkthrough.
- `my-documents/applications/` and `my-documents/reports/` — empty (only `.gitkeep`). Files materialize as skills run.

**What's NOT here (intentionally):**

- No tailored resume / coverletter — `resume-tailor` and `cover-letter` produce these.
- No company-research report — `company-research` produces it (Beat 3).
- No interview-prep brief — `interview-coach` produces it (Beat 7).
- No DOCX / PDF — `scripts/generate-docx.py` produces them (Beat 8).

**How to use:**

1. Create a fresh test folder, e.g. `~/dev/launch-shoot/avery/`.
2. Copy `starter/my-documents/` into the test folder so it lives at `~/dev/launch-shoot/avery/my-documents/`.
3. Launch Claude Code or Cowork in that test folder.
4. Run the walkthrough beats in order. Each one writes into the test folder, not back into this repo.

After the walkthrough, the test folder's `my-documents/` should look like the committed end-state at [`../my-documents/`](../my-documents/). Compare to verify.
